## Analysis

### Hot-path performance (R58-R59 prepared statements)

The 5 prepared statements moved to constructors in R58-R59 are performing
at **0.002-0.006ms per call** (178K-453K ops/sec). This confirms the
optimization is working: better-sqlite3's prepared statement cache lookup
is eliminated, and the Statement object is held directly.

### SWR cache (R37-R50)

The SWR cache fresh hit is **0.0003ms** (3.4M ops/sec) — essentially free.
The cache eliminates 100% of the underlying query cost for fresh entries.
For stale entries, the caller still gets 0ms return (background refresh
defers the cost to setTimeout(0)).

### Bulk queries (R40 optimization)

getBulkEdges for 100 nodes takes **1.13ms** — this replaces what would be
200 individual getNeighbors calls (each ~0.5ms = 100ms total). The bulk
optimization delivers a **88x speedup** for the graph layout endpoint.

### Write path

createNode takes **0.112ms** (8,956 ops/sec) — fast enough for real-time
vault sync. The write path includes INSERT + JSON serialize + sync_state
update + junction table sync, all in one transaction (R46 F7).

### Comparison with V1 C engine

V1's C API direct access to SQLite has ~0.001ms overhead per query
(measured in V1 benchmarks). V2's better-sqlite3 adds ~0.003ms of JS
binding overhead. The difference is **negligible** — both are sub-0.01ms
for single-row lookups. The SWR cache makes V2 **faster** than V1 for
repeated queries (V1 has no application-level cache).

### Verdict

**All 19 benchmarks pass with "excellent" or "good" assessment.**
No operation exceeds 2ms. The hot-path (prepared statements + SWR cache)
is sub-0.01ms. The V2 sidecar is not a performance bottleneck — the
bottleneck is V1's indexation (CPU-bound, takes seconds to minutes).

The R58-R59 prepared statement optimization and R37-R50 SWR cache are
confirmed effective. No further performance optimization is needed for V2.
